What was built and what broke

A concise record of every step taken, every error hit, and exactly how each one was resolved — from first API call to completed sandbox payment.

What I Implemented
End-to-End Payment Integration (Adyen)
  • Set up an Adyen sandbox environment, including merchant account configuration and API credentials, to mirror a real merchant onboarding flow.
  • Built a Node.js backend (Express) endpoint to create Checkout Sessions via Adyen's API, handling authentication, request construction, and response parsing.
  • Used Postman extensively to test and validate API requests before integrating them into the backend, which helped isolate issues like incorrect headers, malformed JSON, and authentication failures early in the process.
  • Integrated Adyen Web Drop-in v6 on the frontend using the client key, implementing the full session-based checkout flow and handling payment lifecycle events (onPaymentCompleted, onError, etc.).
  • Configured allowed origins in Adyen's Customer Area to resolve CORS issues during local development, understanding how client-side authentication ties to domain restrictions.
  • Successfully completed and verified the full sandbox payment lifecycle using Adyen test cards, including handling authorised, refused, and error states.

Errors encountered & fixed
12 real issues resolved during development
401 UnauthorizedAPI key not saved

What happened: Every request to the Sessions API returned a 401 immediately after credential generation. The API key was freshly created in the Adyen Customer Area but the request was still being rejected as if no key existed.

Root cause

Adyen requires you to explicitly click Save after generating an API key — generating the key alone does not persist it. The backend validates requests against the last saved credential state.

Fix

Returned to Developers → API credentials, clicked Save changes, then retried. Authenticated immediately.

Lesson

Always verify Save was clicked after any credential change. This is a common gotcha during initial setup — it does not fail loudly during key generation.

400 / 702Validation error — bad request body

What happened: The Sessions API returned a 702 validation error about invalid or missing fields, even though the request body looked correct.

Root cause

In Postman, the request body was set to form-data rather than raw JSON, and the Content-Type: application/json header was absent, causing Adyen to misparse the payload entirely.

Fix

Switched Body to raw → JSON and added Content-Type: application/json explicitly. Adyen correctly parsed the payload and returned a valid session.

Lesson

Always set Content-Type explicitly for REST APIs — many tools do not set it automatically even when the body looks like JSON.

403 / 901Invalid merchant account

What happened: The Sessions API returned a 901 error — account not found or not permitted — even though the merchant account string appeared correct in the portal.

Root cause

The merchantAccount field was set to the company-level account (e.g. AayushCo) rather than a specific merchant-level ECOM account. Sessions can only be created against individual merchant accounts with a payment channel attached.

Fix

Switched to the correct ecommerce-level account: UCSCAccountECOM. Request succeeded.

Lesson

The Adyen account hierarchy matters: Company → Merchant → Store. Checkout API calls always use the Merchant account (ECOM suffix), not the company-level root.

403 / 010Not allowed — raw card submission blocked

What happened: Attempting to submit raw card details directly via the Payments API returned a 010 error: Not allowed.

Root cause

Raw card data submission is blocked by PCI-DSS scope restrictions. This endpoint requires PCI DSS SAQ D compliance. Adyen prevents merchants from handling raw PANs unless they have formally obtained the right compliance tier.

Fix

Pivoted to the Adyen Web Drop-in with Sessions flow. Drop-in handles card collection inside Adyen-hosted iframes — raw card data never touches my code or server (SAQ A compliant).

Lesson

For most integrations, Drop-in or Components are the correct path. Raw card APIs require a higher PCI level that most merchants do not have.

CORSAllowed origins not configured

What happened: Drop-in failed to initialize — the browser console showed a CORS error when the SDK tried to fetch payment methods. The checkout container stayed blank.

Root cause

Adyen's client-side SDK makes requests from the browser using the client key, which has an allowed origins allowlist. Since http://localhost:3001 was not on the list, every browser request was rejected with a CORS preflight failure.

Fix

In Customer Area → Developers → API credentials → Client-side authentication, added http://localhost:3001 and http://127.0.0.1:3001 to the allowed origins list. Waited ~1–2 minutes for propagation.

Lesson

CORS errors from Adyen's SDK are almost always an origins misconfiguration. The fix is always in the Customer Area — not in the app code.

JS ErrorAdyenCheckout is not defined

What happened: The browser threw ReferenceError: AdyenCheckout is not defined when the initialization code ran. Drop-in never appeared.

Root cause

The original code used the Adyen Web v5 loading pattern, accessing window.AdyenCheckout as a global. In v6, Adyen moved to an ES Module (ESM) architecture — there is no global anymore. The SDK must be imported with named imports inside a type="module" script tag.

Fix

Updated to <script type="module"> and switched to the correct named ESM import from the CDN.

Lesson

v5 → v6 is a breaking change in how Adyen Web is loaded. Always check the SDK version against the loading pattern in use.

NETWORK_ERRORInvalid client key — frontend auth failure

What happened: The checkout UI failed to render and the payment methods section remained blank. The browser console showed a CORS-style error along with a NETWORK_ERROR during session setup, even though the backend was running correctly.

Root cause

The frontend was initialized with an invalid Adyen client key, causing Adyen's session setup request to fail. The browser reported it as a CORS or network error, but the actual issue was a frontend authentication misconfiguration — the SDK was rejected before reaching the payment methods step.

Fix

Replaced the incorrect client key with the valid Adyen test client key from the Customer Area. Refreshed and confirmed the Drop-in loaded successfully.

Lesson

Browser errors like CORS or Failed to fetch can be misleading. When the Drop-in container is blank and the console shows a NETWORK_ERROR, always check the client key and allowed origins before investigating backend connectivity.

Failed to fetchBackend not running — session endpoint unavailable

What happened: The checkout page loaded, but the payment flow could not initialize because the frontend could not reach the session creation endpoint. A toast appeared: "Could not load checkout. Make sure your backend is running on port 3001."

Root cause

The frontend fetch call was pointed at http://localhost:9999/api/sessions — a port with no running service — causing an immediate ERR_CONNECTION_REFUSED. This simulated a backend service being unavailable or misconfigured.

Fix

Restored the correct backend URL to http://localhost:3001/api/sessions and confirmed the Node.js backend was running. Session creation succeeded and the Drop-in rendered normally.

Lesson

Before debugging SDK behavior or frontend logic, first confirm the backend session endpoint is reachable. Many checkout failures start with a missing backend dependency — the catch block should surface this clearly rather than showing a generic error.

400 / ValidationInvalid payload — type mismatch in session request

What happened: The backend was running and reachable, but the frontend failed to initialize with a toast showing "Could not load checkout." The backend terminal logged an Adyen 400 response with a validation error about the amount.value field.

Root cause

The amount.value field in the session request body was sent as a string ("1000") instead of a number (1000). Adyen enforces strict schema types — a string where a number is expected causes the entire session creation to fail with a 400 validation error. The frontend originally mislabelled this as a client key / CORS issue because NETWORK_ERROR was being matched too broadly in the catch block.

Fix

Corrected the backend payload back to value: 1000 (integer). Also updated the frontend catch block to distinguish 400/422 session failures from frontend config errors, so the toast now correctly reads: "The session request payload was rejected. Check backend request fields and types."

Lesson

Adyen's API is strictly typed — always send amount.value as an integer (minor units), not a string. When debugging checkout failures, check backend terminal logs first: a 400 from Adyen with a validation message pinpoints the exact field, whereas the frontend toast alone is often misleading.

403 / 901Missing required field — merchantAccount omitted

What happened: The checkout page loaded, but payment methods failed to initialize and the frontend displayed "Could not load checkout. Please review console logs and configuration." The backend was still running correctly, and the terminal showed Adyen returning a 403 response with error code 901 and the message Invalid Merchant Account.

Root cause

The merchantAccount field was intentionally removed from the backend session request body to simulate a missing required field. Adyen could not associate the request with a valid merchant account, so it rejected it as a security and configuration error rather than a generic validation error.

Fix

Restored the missing field in the session request body: merchantAccount: ADYEN_MERCHANT_ACCOUNT. Restarted the backend and refreshed the frontend, which allowed session creation to succeed again.

Lesson

Required fields do not always fail with generic validation errors. In Adyen's case, omitting merchantAccount results in a merchant account security error (403 / 901). When debugging integrations, backend response codes and messages are much more reliable than frontend toasts for identifying the true cause.

IMPLEMENTATION_ERROREnvironment mismatch — test client key used against live environment

What happened: The checkout page loaded, but payment methods failed to initialize. The frontend displayed "Checkout failed to initialize. Possible frontend configuration issue: verify your Adyen client key and allowed origin." In the browser console, Adyen returned a more specific error: IMPLEMENTATION_ERROR: you are using a test clientKey against the live environment.

Root cause

The frontend was intentionally configured with environment: "live" while still using a test client key and a backend generating test sessions. This created a mismatch between the environment and credentials used in the integration. Adyen enforces strict consistency across environment, client key, and session configuration, and rejected the initialization immediately.

Fix

Restored the frontend configuration to match the backend and client key: environment: "test". After refreshing the page, checkout initialized successfully and payment methods rendered as expected.

Lesson

Adyen requires strict alignment between environment, client key, and backend session configuration. Mixing test and live settings will cause immediate initialization failures. When debugging these issues, browser console errors often provide more precise insight than generic frontend messages, which may misclassify the problem.

UX / DuplicateDuplicate submission — Pay button not disabled after first click

What happened: Nothing visually prevented a user from clicking the Pay button multiple times before the payment completed. In a real product this risks duplicate charges being submitted against the same session. A first fix using dropin.setStatus("loading") caused the UI to get permanently stuck in a loading state, blocking the payment flow entirely.

Root cause

The Drop-in was initialized without any duplicate submission guard. The first attempt used dropin.setStatus("loading") inside onSubmit, but this interfered with Adyen's internal submission lifecycle — the UI locked but the payment could never complete or recover.

Fix

Replaced the setStatus approach with a simple boolean flag: let paymentInProgress = false. On the first click onSubmit sets it to true and shows a processing toast. Any subsequent click while the flag is true shows "Payment already in progress" and returns early without touching the Drop-in state. The flag is reset to false in onPaymentCompleted, onPaymentFailed, and onError so the flow can recover cleanly from any outcome.

Lesson

Let Adyen manage its own UI state — do not force setStatus inside onSubmit unless you fully control the rest of the lifecycle. A lightweight flag is the safer pattern for duplicate prevention: it protects against repeat clicks without interfering with how the Drop-in processes and completes the payment.
